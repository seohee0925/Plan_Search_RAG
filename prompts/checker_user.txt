### Evidence State Summary

### Verified Evidence Units
Write bullet lines in this exact format:
- [EVID_1] answer_key=... | value_hint=... | basis=... | note=...
Rules:
- Emit only evidence that is strong enough to support later projection.
- Final category or case-type projection happens here, not in the retriever.
- Retriever evidence-unit statuses are only collection-time labels, not final semantic judgments.
- Decide here whether each grounded candidate should be used, rejected as off-target, or left unresolved.
- Distinguish strong usable evidence from weak but still relevant evidence.
- For category or case-type tasks, a local formulation can be strong enough if it directly identifies the dispute, claim, benefit, or action that the target category refers to.
- For category or case-type tasks, broad compensation, supervision, liability, or administrative-duty wording is not enough by itself when it still requires an extra semantic leap to reach the target category.
- Repeated weak evidence does not become strong by duplication.
- basis should cite the relevant step ids, packet refs, or evidence units.
- Do not use the pipe character inside field values.

### Projected Answer State
Write bullet lines in this exact format:
- [STATE_1] answer_key=... | value=... | basis=...
Rules:
- This is the first place where answer-level projection happens.
- Keep values inside the allowed answer space.
- Prefer grounded semantic projection over exact lexical matching, but stay conservative when the mapping is weak.
- If a candidate formulation is relevant but still broader than the target category, leave it out of the projected answer state and keep it as ambiguity or a remaining gap.
- For numeric values, normalize the sign and units conservatively.
- Do not use the pipe character inside field values.

### Slot Fill State
Write bullet lines in this exact format:
- [SLOT_1] doc=... | title=... | state=... | basis=... | note=...
Rules:
- For exhaustive closed-set list tasks, every document is a required slot and must appear exactly once here.
- state must be one of: selected, rejected_off_target, rejected_no_evidence_after_scoped_search, unresolved_scope_gap, unresolved_conflict.
- selected means this document should appear in the final projected answer state.
- rejected_off_target means current evidence is enough to say the document is about a different category or dispute than the target.
- rejected_no_evidence_after_scoped_search means the current scoped evidence is sufficient to conclude that no usable evidence supports the target for this document.
- unresolved_scope_gap means the current scoped evidence is not enough yet and more targeted reading may still change the outcome.
- unresolved_conflict means the current evidence for this document conflicts or cannot yet be reconciled safely.
- Do not use rejected_no_evidence_after_scoped_search when the current scope already contains plausible or ambiguous evidence; use unresolved_scope_gap instead.
- When the current scope contains a concrete but different dispute or action that is sufficient to rule out the target, prefer rejected_off_target over rejected_no_evidence_after_scoped_search.
- Prefer terminal states whenever the evidence really closes the document slot.
- Use basis to cite the main step id, evidence unit, or packet refs that justify the slot state.
- Do not use the pipe character inside field values.

### Remaining Gaps
Write bullet lines in this exact format:
- [GAP_1] answer_key=... | issue=... | why=...
If nothing remains, write none.
Rules:
- When the verdict is FALSE, use this section to name only the unresolved slots or unresolved projections that still block a safe final answer.
- Distinguish weak evidence from genuinely missing evidence when you describe the gap.
- Prefer gap descriptions that point to the exact ambiguity, missing link, or over-broad projection that blocks a safe final answer.
- If possible, make it obvious which step, document, or candidate formulation the gap refers to.

### Repair Requests
Write bullet lines in this exact format:
- [REPAIR_1] action=... | target=... | why=...
If no repair is needed, write none.
Rules:
- Prefer concrete repair actions such as widen_scope, revisit_step, compare_conflicting_units, or request_fallback_region_read.
- Name the affected answer key, search target, step id, or doc id when possible.
- Prefer concrete step ids, document ids, or explicit search target names over vague phrases that refer to the whole graph.
- When only part of the execution graph needs work, do not request a whole-graph repair.
- Only request repair for slots whose state is unresolved_scope_gap or unresolved_conflict.
- Each repair request should correspond to one concrete unresolved point from Remaining Gaps.

### Sufficiency Verdict

FALSE

Rules:
- For exhaustive closed-set list tasks, return TRUE when every document slot is in a terminal state and the projected answer state is consistent with the selected slots.
- Return FALSE only when at least one required slot remains unresolved or the current projected answer state is inconsistent with the slot states.
